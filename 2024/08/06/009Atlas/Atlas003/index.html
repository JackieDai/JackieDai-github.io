MatchBox+Atlas启动流程

写在前面

查看文档，已经知道AtlasCore启动AtlasGroupProcess和AtlasDetectionProcess用来分别执行group.lua和detect.lua

AtlasDetectionProcess 的作用主要是根据相应的URL找到Device和Resouce,并将其映射到相对应的Group上，相当于测试的准备工作

而AtlasGroupProcess则控制整个测试周期，

主要就分为4大步骤

  1. 初始化 GSM
  2. 设置Group
  3. 开启测试循环,直到退出循环
  4. 测试结束

个人理解：通过观察log /Library/Logs/Atlas/active/atlas.log 可知，AtlasDetectionProcess 的操作应该是在 【GroupSetup 方法完成之前】完成的

GroupStateMachine : Init

初始化一些函数签名，以及一些初始条件

GroupStateMachine : GroupSetup

主要作用是

  1. 获取GROUP的Resource Group.getResources()
  2. 给GROUP设置插件,这里会调用group.lua中自定义的function groupSetup(resourcesURLs)方法，其内部主要功能
    1. 加载CSV并检查格式
    2. 加载Group 插件，其中最主要的是加载Group所需的插件

通过查看Atlas.log文件发现，Detection在GroupSetup函数执行完成之前已经完成了DEVICE和Resource与Group关联的操作了。

个人理解：

Group是一个序列化测试项和共享资源的容器，那么我们在创建或者说设置组的时候，说明测试项和共享资源均已经准备完毕。这里所说的测试项是指我们的CSV文件，这里的资源包括，治具，产品，插件等。

GroupStateMachine : TestLoopStart

这部分是产品测试逻辑的核心，想象一下，在设备开始测试的时候需要哪些设备？大概需要哪些逻辑？

  1. 我们首先可以确定的是，结束测试循环的条件即groupShouldExit,这个函数会返回boolean 类型的值

    • true == Matchbox will exit from current group script after teardown group plugins,and Atlas2 will start group script again.
    • false == Matchbox will loop inside current group script to next device detection, reusing existing group plugin.

    由上可知，true ,每当group测试完成之后，都会释放资源，而 false 不会。在Matchbox框架中，默认返回的是"false", 当然这个值也可以在Plugins.lua对应的方法中重写.

  2. 接下里，循环体里面可能会需要哪些逻辑呢？

    getSlots

    在groupsetup步骤时，我们已经知道Group的容器已经存在所需的插件了,接下来是去获取 Group中的穴位也即slot（这是我们在 station configuration plist 中 key == SlotConfig中配置的，当前测试站位中一般一个Group 对应 一个slot）;

    fetchAndExcuteDevice

    当获取到slot之后，就应该去寻找并关联当前Group和slot对应的DUT，找到之后，开始loop循环测试项直至不满足循环条件退出(循环的条件用户可自定义,这个条件是侦测设备次数，默认只循环一次)。

    具体的循环体，包含以下步骤

    GroupStateMachine : DeviceSetup

    这里主要做两件事

    • Group.startDevice; Group 启动设备

      这个过程中会命名DeviceName，格式为G=1:S=Slot1(local deviceName = "G=" .. Group.index .. ":S=" .. slot)

    • deviceFT.setup； 主要是加载 设备的 插件

    理解：

    ​ 为执行测试ExecuteTest做准备工作，需要设备标识(即DeviceName),需要操作设备的插件(如，串口插件)

    GroupStateMachine : GroupStart

    这里也是为 ExecuteTest做准备工作，同时允许用户做一些自定义的逻辑操作

    run before each test cycle

    GroupStateMachine : ExecuteTest ****

    这里是整个测试框架的核心部分，主要是调度和执行任务。具体参考源码和官方文档，简要逻辑如下

    1. Group 调度某一个设备Group.scheduler(deviceName)
    2. 将该设备所有的测试项加工好放进任务包 dag中，(这个加工的过程主要是序列化测试项事件会调用SequenceControl.scheduleTests)
    3. Group 去执行 任务包中的任务

    理解：Group 启动之后一定是去执行 DEVICE的 任务的，所以GROUP会根据DeviceName去调度设备，Device要把需要测试的TestCoverage通过GroupSequenceControl序列化放入任务包中(在这里DEVICE做了两个操作scheduleDAG,scheduleFinalDAG将测试项传递给Group)，最后再由Group去执行任务包，也就是执行测试项。

    GroupStateMachine : GroupStop

    run after each test cycle when all slots finish testing

    GroupStateMachine : DeviceTeardown

    Group停止调度DEVICE，释放Device的资源和插件

    • Group.stopDevice

    • deviceFT.teardown for each device

      关闭相应串口，断开相应Socket链接 等操作

GroupStateMachine : GroupTeardown

释放Group资源

总结

纵然有很多启动细节，没法查看，但是根据源码和log分析依然能够觅寻些许蛛丝马迹。

在程序启动过程中，首先是AtlasCore启动，AtlasCore 会去启动AtlasGroupProcess和AtlasDetectionProcess，这两个进程是相对独立的，他们会根据station configuration plist的配置，去分别配置相对应的Group环境，以及侦测对应的DUT和Resources.

在上述过程完成之后，GSM会执行文章所述的四大步骤即：

  • GSM init
  • GSM setUp
  • GSM loop test
  • GSM teardown

GSM setUp 的过程中，主要是Check CSV 的 formatter，确保CSV格式正确，这相当于是预先检查的机制; 另外一个主要的任务是加载Group所需要用的插件，这些插件会贯穿Group的整一个生命周期，直到 teardown 时释放

GSM loopTest 是核心部分，即对DEVICE测试；这里的逻辑主要分为两大部；

  • 第一部分是Group这个容器获取Slots信息，这些信息都是在station configuration plist文件中配置的；

  • 第二部分是DEVICE的生命周期，这里的流程些许复杂。首先给设备命名，Group启动设备，设备在加载自己所需要的插件，这一步骤称之为DeviceSetUp是为执行Device测试做准备;其次是Group start操作，用户在Device开始测试之前的逻辑部分可以写在这里；第三就是核心逻辑部分 "执行测试项"具体过程看上文分析。接下来就是GroupStop(用来编写结束测试时的逻辑部分)以及Device.teardown的操作(释放DEVICE的资源和插件)

    1. Group.getSlots()
    2. Device.setup()
    3. Group.start()
    4. Group.excuteTestItem()
    5. Group.stop()
    6. Device.teardown()
